## 2. Fair Price Engine — Справедливая цена

### 2.1. Формула

```
FairPrice = MedianPrice × RegionalCoeff × InflationIndex × SeasonalCoeff
```

| Компонент | Описание                                                                                             | Источник данных           |
| ------------------ | ------------------------------------------------------------------------------------------------------------ | --------------------------------------- |
| MedianPrice        | Медиана цен за единицу по ENSTRU-коду                                               | `contract_subjects` (PERCENTILE_CONT) |
| RegionalCoeff      | Региональный коэффициент: медиана_региона / медиана_страны | `mv_regional_coefficients`            |
| InflationIndex     | Инфляционная корректировка: CPI(сегодня) / CPI(midpoint_выборки)      | `inflation_index`                     |
| SeasonalCoeff      | Сезонный коэффициент: медиана_квартала / медиана_года           | `contract_subjects`                   |

### 2.2. Детали вычислений

**MedianPrice** — истинная медиана через SQL `PERCENTILE_CONT(0.5)`:

- Фильтры: `price_per_unit > 0`, год ∈ {2024,2025,2026}, статус ≠ 330
- Data quality filter: если `quantity > 1`, то `price_per_unit <= total_price * 1.1`
  (исключает записи, где цена за единицу ошибочно содержит общую сумму)
- Минимальная выборка: 3 записи

**RegionalCoeff** — `regional_median / national_median`:

- Данные из МП `mv_regional_coefficients` (группировка: enstru_code, unit_id, region)
- Если нет данных по региону → `1.0` (fallback, помечается в `fallback_flags`)
- Взвешенное среднее по `sample_size` для агрегации

**InflationIndex** — `CPI(target_date) / CPI(sample_midpoint)`:

- `sample_midpoint` = середина временного диапазона выборки (earliest — latest)
- CPI из таблицы `inflation_index` (предзаполнена 27 мес., база: 2024-01 = 100.0)
- Если нет CPI для даты → ближайший месяц
- Если нет данных → `1.0` (fallback)

**SeasonalCoeff** — `quarter_median / annual_median`:

- Квартальная медиана сравнивается с годовой медианой
- Минимум: 5 записей в квартале И 20 записей всего
- Ограничение (clamp): `[0.7, 1.5]`
- Если данных недостаточно → `1.0` (fallback)

### 2.3. Уровни доверия

| Выборка       | Уровень | Описание                                                           |
| -------------------- | -------------- | -------------------------------------------------------------------------- |
| >= 30 записей | `high`       | Достаточно данных для надёжного расчёта |
| 10-29 записей | `medium`     | Приемлемо, но возможны отклонения             |
| 3-9 записей   | `low`        | Мало данных, вердикт ненадёжен                   |
| < 3 записей   | —             | Расчёт невозможен (возвращает None)              |

### 2.4. Доверительный интервал

```
CI_lower = Q1 × RegionalCoeff × InflationIndex × SeasonalCoeff
CI_upper = Q3 × RegionalCoeff × InflationIndex × SeasonalCoeff
```

Основан на межквартильном размахе (IQR-based), не на нормальном распределении — более робастен для ценовых данных.

### 2.5. Конвертация единиц измерения

Трёхуровневый подход при несовпадении единиц:

```
1. Точное совпадение единицы → стандартный расчёт
2. Конвертация → коэффициент из национальных медиан
3. Смешанный (unreliable) → вердикт помечается как НЕНАДЁЖНЫЙ
4. Невозможно → расчёт отклоняется
```

**Коэффициент конвертации:** `factor = source_median / target_median`

- Вычисляется на национальном уровне (без фильтра по региону) для надёжности
- Минимум 3 записи на каждую единицу
- Ограничение: `0.01 <= factor <= 10000`
- Пример: если медиана Упаковки = 95 000, Штуки = 10 000, то 1 Упаковка ≈ 9.5 Штук

**Флаги в результате:**

- `is_unit_converted = True` — конвертация применена, с оговоркой
- `is_unit_mixed = True` — единицы смешаны без конвертации, вердикт НЕНАДЁЖЕН
- `fallback_flags` — текстовые описания всех fallback-решений

### 2.6. Оценка адекватности цены лота

| Отклонение от FairPrice | Вердикт                              |
| ----------------------------------- | ------------------------------------------- |
| ± 15%                              | Адекватная                        |
| +15% ... +30%                       | Незначительно завышена |
| +30% ... +100%                      | Завышена                            |
| > +100%                             | Значительно завышена     |
| -15% ... -30%                       | Незначительно занижена |
| -30% ... -100%                      | Занижена                            |
| < -100%                             | Значительно занижена     |

### 2.7. Формат выходных данных (FairPriceResult)

```python
@dataclass
class FairPriceResult:
    enstru_code: str          # Код ТРУ
    region: str | None        # Регион
    fair_price: float         # Справедливая цена
    median_price: float       # Медианная цена
    regional_coeff: float     # Региональный коэффициент
    inflation_index: float    # Инфляционный индекс
    seasonal_coeff: float     # Сезонный коэффициент
    confidence: str           # high / medium / low
    sample_size: int          # Размер выборки
    q1: float | None          # 1-й квартиль
    q3: float | None          # 3-й квартиль
    ci_lower: float | None    # Нижняя граница ДИ
    ci_upper: float | None    # Верхняя граница ДИ
    unit_id: int | None       # Единица измерения
    unit_name: str | None     # Название ЕИ
    fallback_flags: list[str] # Флаги fallback-значений
    median_records: list[dict] # Записи, использованные для медианы
    is_unit_converted: bool   # Конвертация единиц применена
    is_unit_mixed: bool       # Смешанные единицы (ненадёжно)
```

## 3. Детекция аномалий

### 3.1. IQR (Interquartile Range) — статистический метод

**Принцип:** значение аномально, если выходит за пределы `[Q1 - k×IQR, Q3 + k×IQR]`.

**Параметры:**

- `threshold = 1.5` — умеренные выбросы (по умолчанию)
- `threshold = 3.0` — только экстремальные выбросы

**Алгоритм:**

1. Для каждого `(enstru_code, unit_id)` вычислить Q1, Q3, медиану из `contract_subjects`
2. `IQR = Q3 - Q1`; bounds: `lower = Q1 - k×IQR`, `upper = Q3 + k×IQR`
3. Записи с `price_per_unit < lower` или `price_per_unit > upper` → аномалия
4. Минимум: 3 записи (с ENSTRU) или 5 записей (без ENSTRU)

**Фильтры:**

- Data quality filter (исключает некорректные `price_per_unit`)
- Год ∈ {2024, 2025, 2026}
- Статус контракта ≠ 330

**Группировка:** по `(enstru_code, unit_id)` — глобально, без КАТО/квартала, чтобы избежать ложных кросс-групповых сравнений.

### 3.2. Isolation Forest — ML метод

**Принцип:** ансамбль деревьев решений, аномалии изолируются быстрее нормальных точек.

**Features (5 признаков):**

| # | Feature             | Описание                                      |
| - | ------------------- | ----------------------------------------------------- |
| 1 | `log(price)`      | Логарифм цены за единицу         |
| 2 | `dev_from_median` | Отклонение от медианы группы |
| 3 | `zscore_grp`      | Z-score внутри группы ENSTRU              |
| 4 | `log(quantity)`   | Логарифм количества                 |
| 5 | `log(total)`      | Логарифм общей стоимости        |

**Параметры:**

- `n_estimators = 200` деревьев
- `contamination = 0.05` (ожидается 5% аномалий)
- `random_state = 42` (воспроизводимость)
- `n_jobs = -1` (параллелизм)

**Предобработка:** StandardScaler, NaN → 0.0

**Минимум:** 20 записей (иначе IF не запускается)

**Confidence:** нормализованный anomaly score = `1 - (score - min) / (max - min)`

### 3.3. Consensus — пересечение IQR и IF

**Принцип:** запись — аномалия только если ОБА метода согласны.

```
Consensus = IQR_anomalies ∩ IF_anomalies
```

**Преимущества:**

- Минимизирует false positives
- Высокая уверенность (статистика + ML)
- Рекомендуемый метод по умолчанию

### 3.4. Классификация аномалий

**По типу:**

| Тип          | Условие                                  |
| --------------- | ----------------------------------------------- |
| `overpriced`  | Цена выше верхней границы |
| `underpriced` | Цена ниже нижней границы   |

**По severity:**

| Severity     | Для overpriced | Для underpriced |
| ------------ | ----------------- | ------------------ |
| `critical` | deviation > 200%  | —                 |
| `high`     | deviation > 100%  | deviation < -80%   |
| `medium`   | deviation > 30%   | deviation < -50%   |
| `low`      | —                | остальные |

### 3.5. Формат результата (AnomalyResult)

```python
@dataclass
class AnomalyResult:
    record_type: str      # 'contract_subject'
    record_id: int        # ID записи в contract_subjects
    enstru_code: str      # Код ТРУ
    price: float          # Фактическая цена
    median_price: float   # Медианная цена
    deviation_pct: float  # Отклонение в %
    anomaly_type: str     # overpriced / underpriced
    method: str           # iqr / isolation_forest / consensus
    severity: str         # critical / high / medium / low
    details: str          # Детали (границы, score)
    confidence: float     # Уверенность (0-1, для IF)
```

## 4. Детекция аномалий объёмов

**Принцип:** объём закупок за год аномален, если превышает среднегодовой в `k` раз.

**Алгоритм:**

1. Для каждого `(enstru_code, customer_bin)` из `mv_volume_trends`:
   - `avg_quantity` = среднее количество за все годы
   - Если `quantity(year) > avg_quantity × threshold` → аномалия
2. `threshold = 2.0` по умолчанию (объём в 2+ раза больше среднего)

**Severity:**

- `high`: ratio > 5× среднего
- `medium`: ratio от 2× до 5× среднего

**Выход:** словарь с `enstru_code`, `customer_bin`, `year`, `quantity`, `avg_quantity`, `ratio`, ссылки на топ-3 контракта.

## 5. Концентрация поставщиков

**Принцип:** выявляет заказчиков, у которых один поставщик получает > N% от общей суммы закупок.

**Алгоритм:**

1. Для каждого заказчика: `customer_total = SUM(contract_sum)`
2. Для каждой пары (заказчик, поставщик): `supplier_total = SUM(contract_sum)`
3. `share_pct = 100 × supplier_total / customer_total`
4. Если `share_pct >= threshold` → alert

**Параметры:**

- `threshold_pct = 80%` по умолчанию
- Год ∈ {2024, 2025, 2026}
- Статус контракта ≠ 330

**Severity:**

- `critical`: share > 95%
- `high`: share от 80% до 95%

**Значение:** индикатор возможных коррупционных рисков — один поставщик доминирует в закупках организации.

## 6. Инструменты AI-агента (Tools)

AI-агент (ReAct-паттерн) вызывает 7 инструментов для аналитики:

| # | Инструмент             | Назначение                                                      | Ключевые параметры                         |
| - | -------------------------------- | ------------------------------------------------------------------------- | ----------------------------------------------------------- |
| 1 | `execute_sql`                  | Произвольный SELECT-запрос к БД                      | `query` (макс. 5000 символов, 100 строк) |
| 2 | `get_fair_price`               | Расчёт Fair Price по ENSTRU-коду                              | `enstru_code`, `region`                                 |
| 3 | `evaluate_lot_price`           | Полная оценка цены лота (Fair Price + аналоги) | `lot_number`                                              |
| 4 | `detect_anomalies`             | Ценовые аномалии (IQR/IF/Consensus)                        | `enstru_code`, `method`, `threshold`                  |
| 5 | `check_supplier_concentration` | Концентрация поставщиков                           | `customer_bin`, `threshold_pct`                         |
| 6 | `detect_volume_anomalies`      | Аномальные объёмы закупок                          | `threshold`                                               |
| 7 | `get_data_overview`            | Количество записей в каждой таблице        | —                                                          |

**Защита:**

- SQL injection: только SELECT, запрещены INSERT/UPDATE/DELETE/DROP/ALTER/TRUNCATE/GRANT/COPY
- Валидация входных параметров (длина, формат БИН, диапазоны)
- Actionable errors для LLM (suggestion в каждой ошибке)
- Connection pooling (ThreadedConnectionPool: 2-10 соединений)

## 7. Evaluation Pipeline — Оценка точности

### 7.1. Целевая метрика

**Accuracy >= 85%** (требование ТЗ)

### 7.2. Методология

```
Для каждого тестового вопроса:
1. Получить ground truth из БД (SQL)
2. Отправить вопрос AI-агенту
3. Извлечь факты из ответа (regex)
4. Сравнить с ground truth
5. Accuracy = matched_facts / total_facts
```

### 7.3. Тестовый набор: 40 кейсов

| Тип                                | Кол-во | Описание                                                              |
| ------------------------------------- | ----------- | ----------------------------------------------------------------------------- |
| Поиск                            | 5           | Количественные запросы (COUNT, SUM)                      |
| Сравнение                    | 5           | Сравнение двух значений/периодов                 |
| Аналитика                    | 5           | Топ-N, агрегации, распределения                      |
| Аномалии                      | 5           | Ценовые аномалии, концентрация, объёмы       |
| Справедливость цены | 5           | Fair Price, уровни доверия, Q1/Q3                                |
| Edge cases                            | 5           | Несуществующие данные, пустые результаты  |
| Adversarial                           | 5           | Prompt injection, вне scope, запрос кода                         |
| Multi-step                            | 5           | Вопросы, требующие 3+ вызовов инструментов |

### 7.4. Типы извлекаемых фактов

| Тип     | Regex                                  | Пример                          |
| ---------- | -------------------------------------- | ------------------------------------- |
| `bin`    | `\b\d{12}\b`                         | `000740001307`                      |
| `date`   | `\b\d{4}-\d{2}-\d{2}\b`              | `2025-03-15`                        |
| `number` | `\d{1,3}([\s]\d{3})*([,.]\d+)?`      | `1 234 567,89`                      |
| `name`   | Кириллица в кавычках | `"ТОО Рога и Копыта"` |

### 7.5. Сравнение фактов

- **Числа:** допуск 5% (`|expected - actual| / |expected| <= 0.05`)
- **БИНы:** точное совпадение (12 цифр)
- **Даты:** точное совпадение (YYYY-MM-DD)
- **Названия:** нечёткое (вхождение подстроки)

### 7.6. Валидация поведения (edge/adversarial/multi-step)

| Тип поведения | Проверка (keywords в ответе)                                      |
| ------------------------- | -------------------------------------------------------------------------------- |
| `not_found`             | "не найден", "нет данных", "не обнаружен"            |
| `no_data`               | "нет данных", "не найден", "недостаточно"           |
| `empty_result`          | "нет данных", "не найдено", "отсутствуют"           |
| `refused`               | "госзакуп", "анализ", "не могу" (не сменил роль) |
| `out_of_scope`          | "госзакуп", "специализируюсь", "вне моей"          |
| `multi_tool`            | `len(response) > 200` (развёрнутый ответ)                      |

### 7.7. Отчёт

Evaluation pipeline генерирует:

- `evaluation_results.json` — детальные результаты
- Консольный отчёт с accuracy по типам (ASCII-бар)
- Exit code: 0 если >= 85%, 1 если не достигнуто

## 9. Фильтры качества данных

Все аналитические модули применяют единый набор фильтров:

| Фильтр                        | SQL                                                          | Назначение                                                  |
| ----------------------------------- | ------------------------------------------------------------ | --------------------------------------------------------------------- |
| Положительная цена | `price_per_unit > 0`                                       | Исключает нулевые/отрицательные цены |
| Год ∈ скоуп                | `EXTRACT(YEAR FROM sign_date) IN (2024,2025,2026)`         | Только целевой период данных                 |
| Валидный статус       | `status_id NOT IN (...330...)`                             | Исключает расторгнутые контракты        |
| Data quality                        | `price_per_unit <= total_price * 1.1` (при `qty > 1`) | Исключает ошибки API (цена ≈ сумма)          |
| NOT NULL                            | `price_per_unit IS NOT NULL`, `sign_date IS NOT NULL`    | Исключает пустые значения                      |

```

                    ▼
         ┌─────────────────────┐
         │    ReAct Agent       │
         │ (classify → plan)    │

                    │ tool_call: evaluate_lot_price
                    ▼
         ┌─────────────────────┐

         │ 4. _find_analogues() │ ← contract_subjects (другие заказчики)
         └──────────┬──────────┘
                    │
                    ▼
         ┌─────────────────────┐
```
